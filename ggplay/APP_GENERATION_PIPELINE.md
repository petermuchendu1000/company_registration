# App Generation Pipeline

This pipeline prepares Android apps and Play Console listing dossiers locally.
It does not interact with Google Play.

## Two-App Quality Gate

Use two companies until the template, metadata, listing assets, signing, and
verification steps are clean:

```powershell
python -m apps.generator.self_check --count 2 --offset 0
python -m apps.generator.generate --count 2 --offset 0 --skip-build
python -m apps.generator.audit --count 2 --offset 0 --require apk
python -m apps.generator.audit --count 2 --offset 0 --require both
```

`self_check` validates the mutation catalog, minimum five screens/functions,
metadata lengths, and template UI wiring without needing the Android SDK.
`--require apk` checks the current local installable artifacts. `--require both`
is the Play-publishing artifact gate and should fail until signed AABs are built.

## Generated Per-App Outputs

Each selected app folder under `pipeline_output/apps/{company_number}/` contains:

- `manifest.json`: app identity, package, artifact names, hashes, and support email.
- `play_listing.json`: Play Console listing fields and app-content answers.
- `play_listing.md`: human-readable listing dossier.
- `data_safety.json`: Data safety answers for a zero-permission, no-collection app.
- `privacy.html`: local privacy policy.
- `icon-512.png`: Play icon candidate.
- `feature-graphic.png`: Play feature graphic candidate.
- `phone-screenshot-1.png` and `phone-screenshot-2.png`: phone screenshot candidates.
- `install.html`, `qr.png`, and `apk.sha256` when an APK exists.
- `signing.json` and `{company_number}.jks` locally only; do not commit these.

## Current Template Positioning

The app is a privacy-first work toolkit:

- No login.
- No ads.
- No Android permissions.
- No analytics SDK.
- No network calls.
- Minimum five selected screens/functions per generated app.
- Local notes, actions, metrics, privacy panel, and recent activity.

This keeps the Play listing, privacy policy, Data safety answers, and manifest
consistent.

## Functionality Mutation Catalog

The generator currently has 12 offline modules. Each app receives at least five,
chosen from SIC-code preferences plus deterministic company-specific ranking:

- Work Log: local session entries with start/finish flow.
- Checklist: routine task or site checks.
- Daily Plan: short local plan for the day.
- Notes: on-device handover or follow-up notes.
- History: recent activity review.
- Insights: local counts and completion summary.
- Handover: end-of-session handover note.
- Inventory Check: equipment, supply, or delivery-material checks.
- Mileage Notes: manually typed trip notes, with no location permission.
- Incident Notes: private local follow-up notes.
- Reference: local reminder cards for repeated steps.
- Support Info: support and privacy details, with no contacts permission.

## Play Publishing Gate

Before a Play upload, each app needs:

- Signed AAB generated by Gradle.
- AAB hash recorded in `manifest.json`.
- APK/AAB verification with Android SDK `aapt2` and `apksigner`.
- Final developer email updated to `dev@registered-domain`.
- Domain-backed privacy policy URL.
- Developer phone number confirmed.
- Human review of screenshots against the real built app.
- Passing Google Play policy gate:

```powershell
python -m apps.generator.policy_gate --count 2 --offset 0
```

## Known Local Tooling Blockers

Sandboxed commands may not be allowed to read the installed Android SDK at:

```text
C:\Users\LENOVO\AppData\Local\Android\Sdk
```

When SDK access is allowed, the two-app build command is:

```powershell
python -m apps.generator.generate --count 2 --offset 0 --artifact both
python -m apps.generator.verify
python -m apps.generator.audit --count 2 --offset 0 --require both
```

The optional Python package `qrcode` is also missing. Until installed, `qr.png`
is generated as a placeholder link card instead of a scannable QR code.
